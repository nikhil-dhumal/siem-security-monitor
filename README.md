# Distributed SIEM System – Security & Anomaly Detector

## Project Information

This project is being developed as part of the **CS682 – Software Engineering** course in the **MTech Computer Science and Engineering program at the Indian Institute of Technology Bombay (IIT Bombay)**.

The goal of this project is to design and implement a simplified **Security Information and Event Management (SIEM)** system that can monitor logs, detect suspicious activity, and help analysts investigate potential security incidents.

This system focuses on clarity, explainability, and educational value rather than production-scale complexity.

# What is a SIEM System?

A **Security Information and Event Management (SIEM)** system is used by organizations to monitor security events occurring across their infrastructure.

Modern systems generate large volumes of logs such as:

- authentication logs
- application access logs
- system activity logs
- network activity logs

A SIEM platform collects these logs, analyzes them, detects suspicious behavior, and generates alerts that security analysts can investigate.

Our project implements a simplified version of such a system to demonstrate how these components work together.

---

# Goal of the System

The primary goal of this project is to simulate how a SIEM platform works internally.

The system will:

- Collect logs from multiple sources
- Normalize and process log events
- Detect suspicious patterns using rule-based detection
- Generate explainable alerts
- Correlate related alerts into incidents
- Provide a dashboard for monitoring and investigation

The design aims to reflect the workflow followed by **Security Operations Centers (SOC)** while keeping the system understandable and easy to experiment with.

# Key Features

## Log Collection

The system collects logs from simulated agents that represent different machines or applications. These logs contain information such as timestamps, users, IP addresses, event types, and system activity.

This allows the system to simulate monitoring activity across a distributed environment.

> Note: Logs used in this project are synthetically generated to simulate real system activity."

---

## Log Normalization

Logs generated by different systems often have different formats. The system converts incoming logs into a **common normalized schema** so that they can be processed consistently.

Normalized fields may include:

- timestamp
- user identity
- source IP address
- event type
- event status
- source system

Normalization allows the detection engine to analyze logs in a consistent way.

---

## Rule-Based Detection Engine

The system includes a rule-based detection engine that identifies suspicious behavior using predefined rules.

Examples of detection rules include:

- repeated login failures within a short time window
- access attempts from multiple IP addresses for the same user
- unusually high request rates
- abnormal activity patterns

When a rule is triggered, the system generates an alert describing the detected behavior.

---

## Explainable Alerts

Each alert generated by the system includes a clear explanation of why it was triggered.

The alert may include:

- the rule that triggered the alert
- the relevant log events
- the time window in which the activity occurred
- the suspected user or IP address

This makes the system easier to understand and helps analysts interpret security events.

---

## Incident Correlation

Security events are often related to each other. For example, multiple alerts involving the same user or IP address may represent a single attack attempt.

The system groups related alerts into **incidents** using attributes such as:

- user identity
- IP address
- application
- time proximity

This allows analysts to investigate related events together rather than treating every alert separately.

---

## Security Analytics

The system will provide basic analytics to help understand security activity in the monitored environment.

Examples of analytics include:

- number of failed login attempts over time
- most common alert types
- most active or suspicious IP addresses
- distribution of incident severity

These insights help analysts identify patterns and potential threats.

---

## Administrative Dashboard

The system provides a web-based dashboard where administrators and analysts can monitor security activity.

Through the dashboard, users can:

- view alerts and incidents
- inspect related logs
- track investigation progress
- analyze security trends

The dashboard will provide interactive views to explore security events.

---

## Incident Lifecycle Management

Alerts and incidents will follow a simple lifecycle that reflects real SOC workflows.

Typical states include:

- Open
- Investigating
- Resolved

This allows analysts to track the progress of investigations.

---

## Audit Logging

All administrative actions performed in the system will be recorded.

Examples include:

- changing alert severity
- marking incidents as resolved
- modifying detection rules

Audit logs provide traceability and accountability for security operations.

---

## Attack Simulation Support

To demonstrate the functionality of the SIEM system, simulated attack scenarios will be generated.

These scenarios may include:

- brute-force login attempts
- abnormal access patterns
- suspicious activity bursts

These simulations allow the detection engine and alert system to be tested and demonstrated.

# Technology Stack

The system is being implemented using commonly used web and data processing technologies.

**Backend**

Python and Django are used to implement the backend services responsible for log ingestion, detection, and incident management.

**Frontend**

React is used to build the web dashboard for monitoring alerts and incidents.

**Data Storage**

PostgreSQL is used to store logs, alerts, incidents, and audit records.

**Messaging and Event Streaming**

Redis is used for message streaming and real-time event communication between system components.

**Containerization**

Docker is used to containerize system components and simulate a distributed deployment environment.

# Educational Focus

This project focuses on building a **clear and understandable SIEM architecture** suitable for learning and experimentation.

Unlike large enterprise SIEM platforms that prioritize large-scale production deployments, this system emphasizes:

- explainable detection logic
- transparent incident reasoning
- simplified SOC workflows
- educational value

The project aims to help students understand how modern security monitoring systems operate internally.
